## Stage 2: Issue Apply, Idempotency, and Orphans
실제 이슈를 생성하고, 반복 실행 시 중복 생성이 없는지(Idempotency), 삭제된 항목이 Orphan으로 탐지되는지 검증한다.

1. **Apply**: `issue_create_from_spec(dryRun: false)` 호출. 생성된 이슈 URL 확인.
2. **Idempotency**: 동일 명령 재호출. "no-op" 또는 "already exists" 상태 확인.
3. **Orphan**: SPEC의 `# 합의 TODO`에서 항목 하나를 삭제 후 `issue_status` 호출. 해당 항목이 `Orphans` 섹션에 나타나는지 확인.

## Stage 3: PR-watch Fake/Golden Boundary
PR watch 기능이 외부 MCP 없이도 저널 라이프사이클을 올바르게 관리하는지 검증한다.

1. `pr_watch_start`로 테스트 PR 등록 (URL 또는 `#123`).
2. `pr_watch_status`로 등록 상태 확인.
3. `pr_event_record`로 가상의 `issue_comment` 이벤트를 저널에 기록.
4. `pr_event_pending`으로 해당 이벤트가 대기열에 있는지 확인.
5. `pr_event_resolve`로 이벤트를 `accepted` 처리 후 저널 반영 확인.

## Stage 4: gh_run Guarded Writes
`gh_run` 도구가 읽기/쓰기 권한을 올바르게 분류하고 가드하는지 검증한다.

1. **Read**: `gh_run({ args: ["issue", "list", "--label", "qa-agent-toolkit"] })` 호출. 즉시 실행 확인.
2. **Write (DryRun)**: `gh_run({ args: ["label", "create", "test-label"], dryRun: true })` 호출. 실행되지 않고 Plan만 출력되는지 확인.
3. **Write (Apply)**: 사용자의 승인 후 `dryRun: false`로 재호출하여 실제 라벨 생성 확인.
4. **Deny**: `gh_run({ args: ["auth", "token"] })` 호출 시 `GhDeniedCommandError` 발생 확인.
